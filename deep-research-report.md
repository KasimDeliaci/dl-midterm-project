## Practical experiment plan for the recommended dataset

For **HAM10000**, I would use a **stratified train/validation/test split**, ideally around **70/15/15** with a fixed random seed, and I would strongly prefer grouping by lesion or metadata identifier if the downloaded metadata exposes near-duplicate lesion relationships. Even if you do not build a full patient-level protocol, the report should state the split policy clearly because medical-image results are sensitive to leakage and split definitions. The dataset size is small enough that this remains easy to manage in Colab, and feature caching will make the later fusion ablations cheap. ŅłĆciteŅłéturn11academia0Ņłéturn18academia2Ņłü

For preprocessing, I would standardize on **224 ├Ś 224 RGB inputs** with ImageNet normalization, because TorchVisionŌĆÖs pretrained ResNet50, MobileNetV2, and EfficientNet-B0 weights are built around 224-pixel classification crops. Use conservative augmentation: horizontal and vertical flips, small rotations, and mild color jitter; avoid aggressive color perturbations that could distort medically relevant lesion appearance. In common TorchVision implementations, the expected pooled feature sizes are **2048** for ResNet50 and **1280** for both MobileNetV2 and EfficientNet-B0, which is exactly why a small shared-dimension projection layer is the cleanest way to implement weighted fusion. ŅłĆciteŅłéturn34view2Ņłéturn31view1Ņłéturn33view1Ņłéturn31view0Ņłéturn32view2Ņłü

For feature caching, I would save **train/val/test tensors separately** for each backbone and each transfer-learning mode. In practice, that means cached files for `frozen/resnet50`, `frozen/mobilenet_v2`, `frozen/efficientnet_b0`, then the same three under `finetuned`. Each cache should store the feature matrix, labels, image identifiers, and the exact resolved config used to produce it. That makes the expensive partŌĆöthe backbone forward passŌĆörun only a few times, while all single, pairwise, and three-way fusion experiments become small MLP jobs.

The report should contain a **main results table** with accuracy, precision, recall, and F1-score for all required ablations; a **single-backbone table**; a **pairwise-fusion table**; a **three-backbone table**; and a **frozen-vs-fine-tuned summary table**. On HAM10000, I would also make **macro-F1** the primary ranking metric in the narrative, even if the assignment only mandates the generic F1-score, because accuracy alone can hide class-specific failures. Add a **confusion matrix** for the best single backbone and the best fused model, a **per-class F1 gain plot** for fusion versus the best single model, and a small **runtime/complexity plot** using training time plus the official TorchVision compute metadata. ŅłĆciteŅłéturn18academia2Ņłéturn18academia3Ņłéturn34view2Ņłéturn31view1Ņłéturn33view1Ņłü

The main risks are predictable. Class imbalance can make accuracy misleading; the mitigation is to foreground macro-F1 and possibly use class-weighted loss for the MLP. Fine-tuning can overfit quickly on a 10k-image medical dataset; the mitigation is to unfreeze only the last meaningful stage or block, use early stopping, and keep the learning rate low. Weighted fusion can become unstable or meaningless if you sum mismatched feature spaces directly; the mitigation is to project all backbone features to the same dimension first, then apply either fixed equal weights or a learnable softmax weighting. Finally, avoid trying to match headline SOTA numbers from papers with different split or preprocessing protocols; compare directionally, not naively.
